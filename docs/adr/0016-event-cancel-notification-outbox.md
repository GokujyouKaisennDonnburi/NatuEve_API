# ADR-0016: イベントキャンセル通知の Transactional Outbox 化と cancel API の非冪等化

- ステータス: Accepted
- 日付: 2026-07-11
- 関連: #125
- 変更: ADR-0012 の決定事項 3（cancel の冪等性）および 4（通知を notification API に任せる）を置き換える（Supersedes, 部分的）

## コンテキストと課題

ADR-0012 では、キャンセルと参加者通知を「フロントが cancel API → notification API を順に呼ぶ」2 ステップ運用とし、cancel API を冪等に設計した。しかし運用を検討した結果、この方式には以下の問題があることが分かった。

1. **通知が保証されない**: cancel 成功後、通知 API 呼び出し前にユーザーがタブを閉じる・通信が切れる等が起きると、参加者は誰も中止を知らないまま放置される。再試行はフロント画面上で失敗を観測できた場合しか機能しない。
2. **「キャンセル済みだが未通知」をサーバーが検知できない**: DB には `cancelled_at` しか残らず、通知したかどうかの記録がない。
3. **業務ルールのクライアント流出**: 「キャンセルしたら参加者に通知する」という不変条件を保証する場所がサーバー側に存在しない（CLAUDE.md「ドメインロジックの集約」に反する）。
4. **再試行による二重送信**: タイムアウト後の再試行で、実際には送信済みのメールが再送されうる。

一方、DB 更新と外部メール API（Resend）呼び出しを単一トランザクションでアトミックにすることは原理的に不可能である（トランザクション内送信はロールバック時にメールを取り消せず、コミット後送信は送信失敗の穴が残る）。

## 決定

**キャンセル確定と「通知の送信予約」を同一 DB トランザクションで永続化し（Transactional Outbox パターン）、バックグラウンドワーカーが送信・自動リトライする。あわせて cancel API を非冪等に変更する。**

1. `event_notification_outbox` テーブルを新設する。
   - `event_id` / `subject` / `body` / `status`（pending / sent / failed）/ `attempts` / `next_attempt_at` / `last_error` / `sent_at` を持つ。
   - 宛先はスナップショットせず、送信時に `event_members` から解決する（ADR-0012 により参加者記録はキャンセル後も保持されるため安全）。
2. `POST /api/v1/events/{id}/cancel` はリクエストボディ `{subject, body}` を必須とする。
   - 検証仕様は既存の一斉送信 API（件名 255 文字以内・本文 10,000 文字以内）と共通化する。
   - `UPDATE events SET cancelled_at = now() WHERE id = $1 AND cancelled_at IS NULL` と outbox への INSERT を同一トランザクションで行う。**両方成功するか、両方成立しないかのどちらかしかない**。
3. cancel API は**非冪等**とする。既にキャンセル済みの場合は `409 Conflict`（`already_cancelled`）を返す。
   - ADR-0012 が冪等を選んだ理由は「フロントの 2 ステップ運用でのタイムアウト再試行の成功判定」だったが、2 ステップ運用の廃止によりこの根拠は消滅した。
   - 非冪等化により「メールは active → cancelled の状態遷移が成立した 1 回だけ予約される」ことが構造的に保証され、二重送信を防げる。
   - 参加申込 API がキャンセル済みイベントに `409`（`event_cancelled`）を返す既存設計ともエラー体系が揃う。
   - フロントは 409 `already_cancelled` を「失敗」ではなく「完了済み」として扱うこと。
4. 送信ワーカー（goroutine 1 本）が outbox を処理する。
   - 30 秒間隔のティッカーに加え、キャンセル成功直後に即時起床（Wake）するため、通常は数秒以内に送信される。
   - 送信失敗時は指数バックオフ（`min(30s × 2^attempts, 1h)`）で自動リトライする。送信試行は**計 8 回（初回 + リトライ 7 回）**までで、上限到達で `status = 'failed'` とし `slog.Error` を出す。
   - Resend 未設定（`RESEND_API_KEY` / `MAIL_FROM` が無い）の場合、ワーカーは起動せず outbox に蓄積される。設定復旧後に自動送信される。
5. 既存の `POST /api/v1/events/{id}/notifications` は変更せず存置する。
   - 任意の運用連絡に加え、outbox が最終失敗（`failed`）した場合の手動再送手段を兼ねる。
6. outbox テーブルは**構造としては汎用**（イベント通知の送信予約）だが、**現時点の書き込み元は cancel フローのみ**とする。`/notifications` の outbox 化は将来の別 ADR で検討する。

## 理由

- **結果整合による保証**: 厳密な「キャンセルとメール送信の同時成功」は分散システムの制約上不可能だが、Outbox により「キャンセルが成立したら通知も必ず最終的に送られる（at-least-once）」を保証できる。サーバークラッシュ後も pending 行が残るため自動回復する。
- **未通知状態の可視化**: 「キャンセル済みだが未送信」が DB のクエリで検知でき、運用でのリカバリが人間の記憶に依存しない。
- **実装コストが規模に見合う**: 単一 API サーバー + PostgreSQL の構成では、外部キューを使わずテーブル 1 つと goroutine 1 本で成立する。

## 影響（結果）

### 良い影響

- イベント中止の通知漏れ（参加者が知らずに当日集合する事故）が、フロント実装や主催者の操作に依存せずサーバー側で防がれる。
- メール基盤の一時障害・レート制限に対して自動リトライで回復する。

### トレードオフ・制約

- **at-least-once の宿命として重複送信がありうる**（送信成功 → `sent` 更新前のクラッシュ時のみ）。中止メールが稀に 2 通届くことは許容する。Resend の Idempotency-Key による抑止は将来課題とする（100 人超イベントのチャンク分割重複の緩和も兼ねる）。
- **フロントの breaking change**: cancel はボディ必須・409 が新たに返る。ADR-0012 時点の冪等な挙動に依存したクライアントは修正が必要。
- ワーカーは単一プロセス前提の実装（claim に `FOR UPDATE SKIP LOCKED` を使わない）。API サーバーを複数インスタンス化する場合は claim 方式の見直しが必要。
- 最終失敗（`failed`）の検知はログ監視に依存する。管理者への能動通知は将来課題。
- **キャンセル通知には宛先数上限ガード（`maxNotificationRecipients` = 100）を適用しない**。任意連絡である一斉送信 API と異なり、中止通知は必須連絡であり参加者数を理由にブロックすべきでないため（意図的な判断）。参加者が 100 人を超えるイベントのキャンセルでは Resend 無料枠（100 通/日）を超過しうる。その場合の送信失敗は試行上限（計 8 回、数時間以内に消化）で `failed` となるため、ログ検知のうえ翌日以降に `/notifications` で手動再送する運用とする。また 100 件超は複数チャンクに分かれるため、後続チャンク失敗後のリトライで先行チャンク分が重複送信されうる（at-least-once の許容範囲内とする）。

## 検討した代替案

- **フロントが cancel → notification を順に呼ぶ（ADR-0012 の決定）**:
  上記コンテキストの問題 1〜4 のとおり、通知の保証がなくクライアント実装に依存する。不採用（本 ADR で置き換え）。
- **cancel API 内で同期的にメール送信（ベストエフォート）+ 失敗時は手動再送**:
  実装は最小だが、「キャンセル成立・通知失敗」の穴が残り、回復が人間の操作（再送 API）頼みになる。不採用。
- **DB トランザクションにメール送信を含める**:
  コミット失敗時に送信済みメールを取り消せず、「キャンセルされていないのに中止メールが届く」最悪の不整合が起きうる。原理的に不成立。不採用。
- **cancel の冪等性を維持し、repository が「今回初めてキャンセルしたか」を返して初回のみ送信**:
  機能的には同等だが、CTE を使う実装が複雑になり、409 を返す非冪等版のほうが意味論（キャンセルは一度きりの状態遷移）が明確でエラー体系も join API と揃う。不採用。
- **外部ジョブキュー（Redis 等）の導入**:
  現構成（VPS + Docker Compose、1 年後に AWS 移行予定）には過剰。PostgreSQL だけで完結する Outbox で十分。不採用。
