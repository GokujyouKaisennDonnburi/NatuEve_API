# ADR-0001: トランザクションメール基盤に Resend を採用

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: #92, ADR-0002, ADR-0003, ADR-0004

## コンテキストと課題

サインアップ確認・パスワードリセット・各種通知など、ユーザー操作に紐づく
トランザクションメールを送信する基盤が必要になった。
選定にあたっては、無料枠の広さ・開発体験（DX）・Go からの扱いやすさ、
そして将来の AWS 移行（ADR: docker.md 参照、Xserver VPS → App Runner/ECS + RDS/Aurora）
との整合を重視した。

## 決定

トランザクションメールの送信基盤に **Resend** を採用する。

## 理由

- **無料枠**: 小規模な立ち上げ期のコストがかからない。
- **DX**: ダッシュボード・ドメイン認証・ログ確認が扱いやすく、導入コストが低い。
- **Go SDK**: 公式に Go SDK があり、本 API（Go + Gin）から素直に組み込める。

## 影響（結果）

### 良い影響
- 実装・ドメイン認証を短期間で立ち上げられる。
- Go SDK 経由で送信処理をアプリ内に閉じられる。

### トレードオフ・制約
- **無料枠は 100 通/日の壁**がある。ユーザー規模が増えるとここが最初のボトルネックになる。
- 規模拡大時は **Resend Pro プランへの引き上げ**、または **Amazon SES への移行**で対応する。
- SES は将来の AWS 移行（ECS/RDS・Aurora）とインフラが整合するため、
  乗り換え先の第一候補として位置づける。送信処理は差し替えやすいよう抽象化しておく。

## 検討した代替案

- **Cloudflare でのメール送信**: 受信（Email Routing）との統合は魅力だが、
  トランザクション送信としての機能・Go からの扱いやすさで Resend に劣ると判断し不採用。
  （受信側での Cloudflare 採用は ADR-0002 を参照）
- **Amazon SES**: 大規模・低単価で AWS 移行とも整合するが、立ち上げ期の DX と
  ドメイン認証・到達性の初期整備コストで Resend に劣る。将来の移行先候補として保持する。
