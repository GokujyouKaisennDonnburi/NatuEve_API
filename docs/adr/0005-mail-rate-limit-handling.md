# ADR-0005: メール送信のレート制限（429）対応方針

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: #94, ADR-0001, ADR-0004

## コンテキストと課題

ADR-0004 により一斉送信は受信者ごとの個別送信で行うため、参加者数に比例して
送信通数が増える。ADR-0001 で採用した Resend にはレート制限（既定 2 リクエスト/秒）と
無料枠の通数制限（100 通/日）があり、送信中に一時的な 429（レート制限）が返る可能性がある。

決めるべきことは次の 3 点。

1. 429 が返ったときにリトライするか、するならどの範囲か
2. リトライしても解消しない場合、API クライアントへ何を返すか
3. 1 回の一斉送信で受け付ける宛先数の上限

## 決定

一斉送信は Resend の Batch API（1 リクエスト最大 100 件）でチャンク送信し、
**429 のみ**指数バックオフでリトライする（最大 4 試行。`Retry-After` /
`ratelimit-reset` ヘッダがあれば待機時間として優先し、1 回の待機は最大 8 秒）。

リトライしても解消しない場合は `service.ErrMailRateLimited` として handler 層へ伝搬し、
クライアントへ **HTTP 429（`rate_limited`）** を返す。

1 回の一斉送信の宛先数上限は **100 件**とし、超過はバリデーションエラー（400）にする。

## 理由

- **429 のみリトライ**: Resend のレート制限はリクエストが処理される前に拒否するため、
  チャンク全体を再送しても重複送信にならない。一方、認証エラーやドメイン未検証などの
  エラーは待っても解消しない恒久的な失敗なので、即座に失敗させる方が原因究明が早い。
- **クライアントへ 429**: レート制限は一時的な状態であり、サーバー内部の異常（500）と
  区別して「時間をおいて再試行すれば成功する」ことを伝えるのが正確。
- **上限 100 件**: Batch API の 1 リクエスト上限（100 件）と一致させることで、
  1 回の一斉送信が常に 1 チャンクで完結し、途中チャンク失敗時に送信済み件数と
  レスポンスの `SentCount`/`FailedCount` がずれる問題を構造的に回避できる。
  無料枠（100 通/日）を 1 回の操作で使い切らないガードも兼ねる。

## 影響（結果）

### 良い影響
- 一時的なレート制限は自動リトライで吸収され、利用者の操作なしに成功する場合が増える。
- 恒久的エラー（設定ミス等）は即座に 500 + サーバーログに真因が残り、調査しやすい。
- 送信結果の件数報告が実態とずれない。

### トレードオフ・制約
- リトライ待機中はリクエストが最長十数秒ブロックする（同期送信のため）。
- 宛先上限を 100 より大きくする場合は、チャンクが複数回に分かれるため、
  チャンク単位の成否集計（部分成功の扱い）を設計し直す必要がある
  （`internal/service/event_notification.go` の `maxNotificationRecipients` のコメント参照）。

## 検討した代替案

- **キュー化して非同期送信**: レート制限を送信側でならせるが、ジョブ基盤・状態管理・
  結果通知の仕組みが必要になり、現段階（宛先最大 100 件・単発操作）には過剰。
  宛先上限を引き上げる際に再検討する。
- **429 も含めて全エラーをリトライ**: 恒久的エラーまで無駄に待たせ、真因の発見が遅れる。
  重複送信リスクの評価もエラー種別ごとに必要になるため不採用。
- **レート制限をクライアントへ 500 で返す**: 「再試行すれば成功する」一時的状態と
  サーバー異常の区別がつかず、クライアントの適切な再試行判断を妨げるため不採用。
