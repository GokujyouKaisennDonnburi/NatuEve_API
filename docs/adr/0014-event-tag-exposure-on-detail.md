# ADR-0014: イベント詳細レスポンスでのタグ公開設計

- ステータス: Accepted
- 日付: 2026-07-11
- 関連: #128, ADR-0013

## コンテキストと課題

ADR-0013 でイベント投稿時のタグ紐づけ（書き込み側）を実装したが、読み取り側は対象外として
「イベント詳細・一覧レスポンスへのタグ表示（`GetByID` 等）は別途対応する」と明示的に保留していた。
本 ADR はその読み取り側を扱う。紐づいたタグをクライアントが取得できるようにする（#128）。

中間テーブル `event_tags`（`event_id`, `tag_id` の複合 PK）と `tags`（`id`, `name`,
`normalized_name`）は既にあり、タグの公開用 DTO `model.TagResponse`（`id`, `name` のみ）も
タグ一覧 API 用に定義済み。

決めるべき点は以下だった。

1. タグをどのエンドポイントで返すか（一覧 `GET /events` のサマリか、詳細 `GET /events/{id}` か、両方か）。
2. 返すタグの表現をどうするか（`normalized_name` を含む詳細形か、`id`+`name` のサマリ形か）。
3. タグの取得を既存の詳細取得処理（`GetByID`）とどう整合させるか。

## 決定

**紐づくタグは詳細エンドポイント `GET /api/v1/events/{id}` のレスポンス（`EventResponse`）に
のみ、`tags`（`TagResponse` = `id`+`name` の配列・`name` 昇順）として含める。一覧サマリ
（`EventSummary`）には含めない。取得は `GetByID` 内で既存の costs / items / images / pdfs と
同じ逐次クエリ方式で行う。**

1. `EventResponse` に `Tags []TagResponse` フィールド（`json:"tags"`）を追加する。
   一覧の `EventSummary` は変更しない。
2. 返す型は既存の `TagResponse`（`id`+`name`）を再利用する。`normalized_name`（内部の
   検索・重複判定用の正規化名）はレスポンスに含めない。
3. `GetByID` に `event_tags et JOIN tags t ON t.id = et.tag_id WHERE et.event_id = $1
   ORDER BY t.name ASC` を追加し、costs / items / images / pdfs と同じく逐次クエリで積む。
   - `e.Tags = []model.TagResponse{}` で初期化し、0 件時は `null` ではなく `[]` を返す
     （既存フィールドの JSON 安定化パターンに合わせる）。
   - 本対応に合わせ、`GetByID` の全サブクエリループ末尾に `rows.Err()` 検査を追加し、
     `ListSummaries` / `SearchSummaries` と流儀を統一した。

## 理由

- **詳細エンドポイントにのみ載せる**のは、#128 のフロント要件が「詳細のみ」であり、一覧サマリに
  タグを載せると 1 リクエストで N 件分のタグ集約（`event_id IN (...)` の集約クエリ、または
  N+1）が必要になり、コストに見合わないため。一覧でタグチップ表示が必要になった時点で
  別途対応できるよう、今回は最小スコープに留める。
- **`id`+`name` のサマリ形（`TagResponse`）を再利用する**のは、`normalized_name` は
  検索・重複判定という内部用途の値で、クライアントに公開する意味がないため。既存のタグ一覧 API と
  表現を揃え、型の重複も避ける。
- **既存の逐次クエリ方式に合わせる**のは、`GetByID` が既に costs / items / images / pdfs を
  それぞれ個別クエリで積む構造になっており、タグだけ本体クエリへ JOIN して集約すると流儀が割れて
  可読性が落ちるため。将来まとめてバッチ化・単一クエリ化する余地は残しつつ、今回は一貫性を優先する。
- **`ORDER BY t.name ASC`** は、表示順を安定させ、レスポンスとテストのアサーションを決定的に
  するため。

## 影響（結果）

### 良い影響

- フロントは詳細取得の 1 レスポンスで紐づくタグを受け取れる（追加の API 呼び出し不要）。
- 表現がタグ一覧 API と揃い、`normalized_name` のような内部値が漏れない。
- 0 件時も空配列を返すため、クライアントの null 分岐が不要。
- 副次的に `GetByID` の全ループが `rows.Err()` を検査するようになり、イテレーション途中の
  エラーで結果が黙って切り詰められる不具合を防げる。

### トレードオフ・制約

- **N+1 / ラウンドトリップ増**: `GetByID` は本体クエリ＋サブクエリ 5 本（costs / items /
  images / pdfs / tags）の計 6 往復になる。単一レコード取得の詳細エンドポイントでは許容範囲だが、
  将来ホットパス化した場合はまとめて集約・バッチ化する候補。
- **並び順の collation 依存**: `ORDER BY t.name` の順序は DB のロケール／collation に依存する。
  現行の Docker（`postgres:17-alpine`）ではバイト順で意図どおりだが、本番移行先（AWS RDS/Aurora）で
  collation が異なると日本語タグの並び順が変わり得る。厳密な並び順が要件化した場合は
  `COLLATE` の明示を検討する。
- **一覧は非対応**: 一覧サマリにタグは出さない。一覧でのタグ表示・タグ絞り込みは別 issue で扱う。

## 検討した代替案

- **一覧サマリ（`EventSummary`）にもタグを含める**: 1 リクエストで N 件分のタグ集約が必要になり、
  N+1 か集約クエリの追加コストが生じる。現行のフロント要件は詳細のみで、費用対効果が合わないため不採用。
  必要になれば後続 issue で追加する。
- **タグ取得専用エンドポイント（`GET /api/v1/events/{id}/tags`）を新設する**: 詳細表示のたびに
  リクエストが 1 往復増える。関連情報を 1 レスポンスに集約する既存の詳細エンドポイント方針
  （costs / items 等）に反するため不採用。
- **本体クエリにタグを JOIN し `json_agg` 等で単一クエリに集約する**: 往復は減るが、既存の
  costs / items / images / pdfs が逐次クエリ方式のため `GetByID` 内で流儀が割れる。今回は
  一貫性を優先し不採用（将来、全サブクエリをまとめて最適化する際に再検討する）。
- **`normalized_name` も含めた詳細形で返す**: 内部の正規化名を公開する必要がなく、レスポンスを
  肥大化させるだけのため不採用。`TagResponse`（`id`+`name`）に統一する。
