# ADR-0006: resend-go SDK の User-Agent を独自値に上書きする

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: #94, ADR-0001, ADR-0005

## コンテキストと課題

一斉送信 API の開発中（2026-07-07）、送信リクエストがすべて 429（`rate_limit_exceeded`）で
失敗する事象が発生した。ただし通常のレート制限とは挙動が異なっていた。

- 429 応答に `ratelimit-limit` / `ratelimit-remaining` / `retry-after` 等のヘッダが一切ない
- 拒否されたリクエストが Resend ダッシュボードの Logs に記録されず、API キーの
  Last used も更新されない（= Resend のアプリケーション層に到達していない）
- ADR-0005 の指数バックオフでリトライしても解消しない

切り分けの結果、**User-Agent 文字列だけで結果が変わる**ことを確認した。
同一コンテナ・同一 API キー・同一ボディで:

| User-Agent | 結果 |
|---|---|
| `resend-go/2.28.0`（SDK デフォルト） | 429 |
| `resend-go/2.27.0`, `resend-go/2.13.0` | 429 |
| `resend-node/4.0.0`, `natueve-api/1.0`, wget デフォルト | 200（送信成功） |

つまり Resend の edge/WAF が `resend-go/*` の User-Agent を持つリクエストを
一律拒否しており、公式 Go SDK が事実上使えない状態だった（Resend 側の不具合とみられる）。

## 決定

resend-go SDK の公開フィールド `Client.UserAgent` を独自値 **`natueve-api/1.0`** に
上書きして送信する（`internal/mail/resend.go` の `mailUserAgent` 定数）。

あわせて本事象は再現手順を添えて Resend サポートへ報告し、根本解決は Resend 側の
修正に委ねる。

## 理由

- 自前の API キーで正規に認証した送信であり、SDK が公式に提供している設定項目を
  使った回避のため、規約・技術の両面で問題がない。
- SDK を差し替えずに 1 行で解消でき、リトライ・エラーハンドリング（ADR-0005）などの
  既存実装をそのまま維持できる。
- Resend 側の修正を待つ選択肢では、修正時期が不明なまま一斉送信機能が停止し続ける。

## 影響（結果）

### 良い影響
- 一斉送信が復旧し、Resend 側の WAF ルール変更の影響を受けにくくなる。
- UA がサービス固有名になるため、Resend のログ上で自サービスのリクエストを識別しやすい。

### トレードオフ・制約
- Resend が SDK の UA を前提にした互換性制御・集計を行っている場合、その恩恵を受けない。
- Resend 側で WAF が修正されれば本来この上書きは不要になるが、識別性の利点があるため
  修正後も維持してよい（撤去は必須ではない）。
- 429 の真因が「本物のレート制限」か「WAF 誤検知」かはヘッダの有無でしか区別できない。
  同種の事象が再発した場合は、`ratelimit-*` ヘッダが空かどうかをまず確認する。

## 検討した代替案

- **Resend 側の修正を待つ**: 修正時期が不明で、その間一斉送信が全断するため不採用。
- **SDK をやめて HTTP API を直接叩く**: UA は自由になるが、リクエスト構築・エラー分類を
  自前実装することになり、SDK の 1 フィールド上書きで足りる問題への対処として過剰。
- **別言語 SDK 相当の UA（例: `resend-node/*`）を偽装する**: 通ることは確認したが、
  実態と異なるクライアント申告は Resend 側の調査・集計を混乱させる。独自 UA の方が誠実で
  識別性も高いため不採用。
