# ADR-0009: 参加状態ログ追加のイベント不存在は 404 で返す

- ステータス: Accepted
- 日付: 2026-07-09
- 関連: #111, ADR-0008

## コンテキストと課題

`POST /api/v1/events/{id}/participation-logs`（#111、ログインユーザーの
参加状態 join/leave を `event_participation_logs` に追記するAPI）を追加した。
このエンドポイントで**指定イベントが存在しない場合に何を返すか**を決める必要がある。

コードベースには ADR-0008 で整理された 2 つの流儀が併存している。

- **認可系（`requireEventOwner` 経由）= 400 (`invalid_request`)**
  `POST /events/{id}/notifications`、`POST /reports`、`GET /events/{id}/members`。
  所有者チェックの延長で「不存在」と「権限なし」を区別せず 400 に倒す。
- **存在参照系（認可チェックなし）= 404 (`not_found`)**
  `POST /events/{id}/join`。匿名参加可で所有者チェックを通さず、
  `repository.ErrEventNotFound` を直接 404 に変換する。

participation-logs はこの 2 分類のどちらにも完全には当てはまらない**新しいケース**である。

- `join` と違い**認証必須**（`profile_id` が NOT NULL のため匿名参加不可）。
- しかし `notifications` / `reports` / `members` と違い、**イベント所有者チェックは無い**。
  ログインユーザーは誰でも自分の参加状態を記録でき、そのイベントの所有者である必要はない。

ADR-0008 は 404 側の根拠を「Join は匿名参加を許容する性質上、認可チェックを通さない」と
述べており、認証の有無ではなく**認可（所有者/権限）チェックの有無**が 400/404 の分岐軸で
あることを示唆しているが、認証ありかつ認可なしの事例は当時存在しなかった。

## 決定

**participation-logs のイベント不存在は HTTP 404 (`not_found`) で返す。**

400/404 の分岐軸は「**認証の有無**」ではなく「**所有者/権限に対する認可チェックの有無**」
であることを ADR-0008 の枠組みの明確化として確認する。

- 認可（`requireEventOwner`）を伴う操作 → 不存在は 400（不存在と権限なしを統合）
- 認可を伴わない操作 → 不存在は純粋なリソースミスなので 404（認証の有無は問わない）

participation-logs は認証は必須だが所有者/権限チェックを持たないため後者に属し、
`repository.ErrEventNotFound` → `service.NotFoundError` → **404** の経路を採る
（`requireEventOwner` は経由しない）。

レスポンスボディは統一フォーマット:

```json
{ "error": { "code": "not_found", "message": "イベントが見つかりません" } }
```

## 理由

- **400/404 を分けている本来の目的は「存在漏洩の防止」であり、これは認可の有無で決まる**。
  400 側は所有者チェックがあるため「イベントが存在するか否か」を非所有者に漏らさないよう
  不存在と権限なしを統合している。participation-logs には守るべき所有者境界が無く、
  不存在を秘匿する動機が無い。したがって存在漏洩を理由に 400 へ倒す必然性が無い。
- **意味論的に 404 が正しい**。作成対象の親リソース（イベント）が存在しないのだから、
  「リソース不存在 = 404」という REST の原則に素直に従える。
- **最も近い類似エンドポイント `join` と一致する**。どちらも `POST /events/{id}/...` で
  イベント配下にレコードを追記し、所有者チェックを持たない。挙動を揃えることで
  クライアントの実装が単純になる。
- **分岐軸を「認可の有無」と明文化することで、将来の判断がブレない**。認証必須という
  理由だけで 400 に寄せると、ADR-0008 が排除したかった「エンドポイントごとの場当たり的な
  分岐」が復活する。

## 影響（結果）

### 良い影響

- 認証の有無ではなく認可の有無で 400/404 を判断する、という明確な基準が言語化され、
  今後エンドポイントを追加する際の判断コストが下がる。
- `join` と同じ 404 挙動になり、イベント配下へ追記する POST 系の挙動が揃う。
- handler の分岐が `event_join.go` の Join と同一パターンに収まり、レビューが容易。

### トレードオフ・制約

- コードベースの 400 / 404 併存は ADR-0008 に続き**継続する**。本 ADR は解消せず、
  むしろ「認可の有無」という軸でその併存を正当化・明確化するもの。
- 将来 participation-logs に「イベント所有者のみ操作可」等の認可要件が加わった場合、
  この決定は再検討が必要になる（その時は `requireEventOwner` 系＝400 側へ移る可能性がある）。
- 全エンドポイントを 404 に統一する方針を採るなら、ADR-0008 も含めて別 ADR で
  段階移行を設計する必要がある。

## 検討した代替案

- **400 (`invalid_request`) に揃える**: 「認証必須エンドポイントは 400 系」という
  括り方も可能だが、400/404 の本来の分岐軸（認可＝存在漏洩の有無）とズレる。
  存在漏洩の懸念が無い participation-logs を 400 にする積極的な理由が無く、
  意味論的にも `join` との一貫性でも劣るため不採用。
- **`requireEventOwner` を通して所有者チェックを課す**: participation-logs は
  参加者本人が自分の状態を記録する操作であり、イベント所有者に限定する要件が無い。
  過剰な認可はユースケースを壊すため不採用。
