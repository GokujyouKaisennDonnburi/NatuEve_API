# ADR-0015: イベント一覧レスポンスでのタグ公開設計

- ステータス: Accepted
- 日付: 2026-07-11
- 関連: #133, ADR-0014, ADR-0013

## コンテキストと課題

ADR-0014 で紐づくタグを詳細エンドポイント `GET /api/v1/events/{id}` にのみ公開し、一覧サマリ
（`EventSummary`）には「N+1 / 集約コストに見合わない」として載せず、必要になれば別 issue で扱うと
明示的に保留していた。本 ADR はその後続（#133）で、一覧でのタグ表示要件が生じたため一覧サマリへの
タグ公開を扱う。

中間テーブル `event_tags`（`event_id`, `tag_id` の複合 PK）と `tags`（`id`, `name`,
`normalized_name`）、公開用 DTO `model.TagResponse`（`id`+`name`）は既にある。一覧は
`ListSummaries`（通常一覧）と `SearchSummaries`（キーワード検索）の 2 経路が `[]EventSummary` を返す。

決めるべき点は以下だった。

1. 一覧で返すタグの表現をどうするか（ID のみか、`id`+`name` か）。
2. 詳細（`GetByID`）が懸念していた N+1 / 集約コストをどう回避するか。
3. タグが 0 件のイベントのレスポンス表現をどうするか。

## 決定

**一覧サマリ（`EventSummary`）に `tags`（`TagResponse` = `id`+`name` の配列・`name` 昇順・
`json:"tags,omitempty"`）を追加する。タグは、サマリ取得後に `event_tags JOIN tags` を
`WHERE et.event_id IN (...)` の 1 クエリで一括取得し、Go 側で各イベントへ割り当てる（N+1 回避）。
`ListSummaries` / `SearchSummaries` の両経路に適用する。タグ 0 件のイベントは `Tags` を nil のまま
とし、`omitempty` で JSON からキーごと省略する。**

1. 返す型は詳細エンドポイントと同じ `TagResponse`（`id`+`name`）を再利用する。`normalized_name`
   は内部の検索・重複判定用のため公開しない。
2. `EventSummary` に `Tags []TagResponse` フィールド（`json:"tags,omitempty"`）を追加する。
3. `attachTagsToSummaries` ヘルパーを追加し、サマリ件数ぶんのプレースホルダ（`$1, $2, …`）を
   動的生成した `WHERE et.event_id IN (...)` で 1 クエリ取得 → `map[event_id][]TagResponse` に
   集約 → 各サマリへ割り当てる。`ListSummaries` / `SearchSummaries` の両方から、`rows.Err()`
   検査後・return 直前で呼ぶ。
4. タグ無しは `Tags` を nil のままにし、`omitempty` で省略する（詳細 `EventResponse.Tags` が
   空配列 `[]` を強制するのとは意図的に異なる）。

## 理由

- **`id`+`name` を返す**のは、一覧でタグチップ（名前）を表示する要件のため。ID のみだとフロントが
  名前解決に追加リクエストかクライアント側マップを要し、一覧表示のたびに往復かキャッシュ管理が発生する。
  `tags` に `name` を含めれば 1 レスポンスで描画でき、詳細エンドポイントの `tags` とも表現が揃う。
  `tags` テーブルは PK 結合のため JOIN コストはごくわずかで、サーバー負荷増は無視できる。
- **`IN (...)` の一括取得**を採るのは、ADR-0014 が一覧保留の根拠とした N+1 を回避するため。
  イベント 1 件ごとにタグを引くと N 往復になるが、一括取得なら往復は 1 回追加で済む。プレースホルダ
  番号のみを動的生成し値は必ず引数経由で渡すため、pgx の配列エンコードに依存せず SQL インジェクションも
  発生しない（既存 `buildSearchWhere` と同じ流儀）。
- **`omitempty` で省略**するのは、一覧では要素数が多く、タグ無しイベントで `"tags":[]` を並べるより
  キーを省く方がペイロードが軽く、クライアントの「タグの有無」分岐も自然になるため。詳細は単一レコードで
  null 分岐を避ける狙いから空配列を返しており、用途の違いに応じて表現を分ける。
- **`ORDER BY t.name ASC`** は表示順とテストアサーションを決定的にするため（詳細と同じ方針）。

## 影響（結果）

### 良い影響

- 一覧の 1 レスポンスで各イベントのタグ（名前つき）を受け取れ、フロントは追加リクエストなしでタグチップを
  描画できる。表現は詳細エンドポイント・タグ一覧 API と揃う。
- ADR-0014 が懸念した N+1 を、往復 1 回追加の一括取得で回避できた。
- タグ無しは `omitempty` でキーごと省略され、一覧ペイロードが不要な空配列で膨らまない。

### トレードオフ・制約

- **往復 1 回追加**: 一覧は本体クエリ＋タグ一括取得の 2 往復になる。N+1 ではないため許容範囲。将来
  さらに最適化するなら本体クエリへの `json_agg` 集約も候補だが、既存の逐次スタイルとの一貫性を優先した。
- **並び順の collation 依存**: `ORDER BY t.name` は DB のロケール／collation 依存（ADR-0014 と同じ制約）。
  本番移行先（AWS RDS/Aurora）で日本語タグの並びが変わり得る。厳密な順序が要件化したら `COLLATE` 明示を検討。
- **詳細と一覧で 0 件時の表現が異なる**: 詳細は `[]`、一覧は省略。クライアントは両者を意識する必要がある
  （一覧では `tags` 未定義＝タグ無しとして扱う）。

## 検討した代替案

- **一覧でも ID のみ（`tagIds []string`）を返す**: サーバーは `tags` テーブルへの JOIN を省けるが、
  名前表示にフロントの追加解決が必要になり、システム全体では往復かキャッシュ管理のコストが増える。表示要件に
  対して非効率で、詳細との表現も割れるため不採用。
- **イベントごとに個別クエリでタグを引く（N+1）**: 実装は素直だが ADR-0014 が保留した当のコストが
  そのまま出る。一覧は N 件を返すため往復が線形に増える。不採用。
- **本体クエリにタグを JOIN し `json_agg` で単一クエリに集約する**: 往復は 1 回に減るが、`ListSummaries`
  の本体クエリはソート順ごとのホワイトリスト文字列で構成されており、そこへ集約を差し込むと全パターンに
  手が入り可読性が落ちる。今回は一括取得の別クエリで一貫性と可読性を優先し、将来の最適化余地として残す。
- **一覧でも空配列 `[]` を返す（詳細と統一）**: 一覧は要素数が多く、タグ無しの空配列がペイロードを膨らませる。
  `omitempty` で省略する方が軽く、#133 の「タグが無い時は omitempty」という要件にも合うため不採用。
