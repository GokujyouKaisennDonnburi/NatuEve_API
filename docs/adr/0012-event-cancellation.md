# ADR-0012: イベントキャンセル（開催取りやめ）の状態管理と API 設計

- ステータス: Accepted（決定事項 3・4 は [ADR-0016](./0016-event-cancel-notification-outbox.md) により Superseded）
- 日付: 2026-07-10
- 関連: #118

## コンテキストと課題

イベント主催者が開催を取りやめるケースに対応する必要が生じた（#118）。
このとき、以下を同時に満たす設計が必要だった。

1. イベントが開催取りやめになったことを一覧・詳細で表現できるようにする。
2. キャンセル前に参加申込済みのユーザー記録（`event_members`）は失わせない。
   主催者が後から参加者一覧を確認できる必要がある。
3. キャンセル後の新規参加申込を防ぐ。
4. キャンセル確定後、参加者へ連絡を取れるようにする。
   ただし通知送信は外部メール基盤（Resend）を使うため、DB 更新と同じトランザクションには巻き込めない。
5. フロント側が「キャンセル API → 通知 API」の 2 ステップを安全に呼び出せるようにする。

## 決定

**イベントのキャンセル状態は `events.cancelled_at TIMESTAMPTZ` で表現し、以下の API 振る舞いを採用する。**

1. `events` テーブルに `cancelled_at TIMESTAMPTZ`（NULL 許可）を追加する。
   - NULL: 開催予定
   - 非 NULL: その時刻に開催取りやめとなった
2. `POST /api/v1/events/{id}/cancel` を新設し、イベント主催者のみ実行可能にする。
   - 認可は `service.requireEventOwner`（ADR-0008）を流用する。
3. cancel API は**冪等**に実装する。
   - 既にキャンセル済みのイベントに対して再実行しても `200 OK` を返す。
   - フロントが cancel 成功後に notification API を呼ぶ 2 ステップ運用で、
     timeout 再試行時の成功判定をブレさせないため。
4. キャンセル時の参加者通知は、既存の `POST /api/v1/events/{id}/notifications` に任せる。
   - cancel API 側はメール送信を行わず、Mailer 基盤への依存を増やさない。
5. キャンセル済みイベントへの新規参加申込（`POST /api/v1/events/{id}/join`）は
   `409 Conflict`（`event_cancelled`）で拒否する。
6. 既存の `event_members` レコードは削除せず、そのまま残す。
7. イベント一覧・詳細でもキャンセル済みイベントを非表示にせず、`cancelledAt` フィールドを返す。
   フロント側で「キャンセル済み」表示を制御する。

## 理由

- **`cancelled_at` は今回の要件に対して最もシンプル**。
  「開催取りやめ」という意味が明確で、監査性も高い。`status` カラムに比べて拡張余地は少ないが、
  今回は「再開 API を作らない」という制約があり、過剰設計を避けられる。
- **通知は別 API に任せることで整合性と責務分離を両立**。
  Resend は外部 API 呼び出しのため、DB トランザクションに巻き込むとロールバックできない。
  cancel を「DB 更新のみ」の冪等 API にすることで、cancel 成功 → notification 失敗のケースでも
  イベントのキャンセル状態は確実に永続化される。
- **`event_members` を残すことで、主催者はキャンセル後も参加者一覧を確認できる**。
  参加者への返金・連絡先確認などの運用に必要。
- **一覧/詳細で非表示にしない**ことで、キャンセル済みイベントへの直リンクや検索結果からの
  アクセスが 404 にならず、フロントが理由を表示できる。

## 影響（結果）

### 良い影響

- フロントは `cancelledAt` の有無だけで UI を切り替えられる。
- 主催者権限チェックが `requireEventOwner` に集約されているため、
  cancel API の認可ロジックも他の主催者限定 API と同じ fail-closed 挙動になる。
- キャンセル済みイベントへの参加申込を DB トランザクション内でロック後に拒否するため、
  並行申込での競合も起きない。

### トレードオフ・制約

- 既存のマイグレーションファイル `20260622140000_create_events.sql` を直接修正した。
  適用済みの DB 環境では `goose reset` または手動 `ALTER TABLE` が必要になる。
- 一覧取得でキャンセル済みイベントも返すため、フロント側で非表示または
  「キャンセル済み」ラベル表示を行う必要がある。
- 今回は「再開（キャンセル取り消し）API」を作らない。将来的に必要になった場合は、
  `cancelled_at` を NULL に戻す更新 API を別途検討する。

## 検討した代替案

- **`status` カラム（`open`/`cancelled` 等）を追加する**: 
  将来の状態拡張（draft/closed 等）に備えられるが、今回の要件には過剰。
  また `cancelled_at` ほど時刻情報を自然に持てない。不採用。
- **`deleted_at` で論理削除を表現する**: 
  「開催取りやめ」と「削除」は意味が微妙に異なる。`deleted_at` だとレコードが
  消去されたように見え、主催者が後から参加者を確認する運用と齟齬が出る。不採用。
- **イベントを物理削除する**: 
  参加者記録・レポート・画像等の関連データも失われるため、不採用。
- **cancel API 内で参加者へ自動通知メールを送る**: 
  Resend 呼び出しをトランザクションに含められないため、送信失敗時に
  「キャンセルは成功したが通知は届かない」という中途半端な状態になりやすい。
  フロント側で notification API を呼ぶ方が、失敗時の再試行と状態の見通しが良い。不採用。
- **キャンセル済みイベントを一覧/詳細から非表示にする**: 
  主催者や参加者が「存在していたイベント」にアクセスできなくなり、
  運用時の問い合わせ対応が困難になる。不採用。
