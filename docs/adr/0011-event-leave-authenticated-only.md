# ADR-0011: イベント参加キャンセルはログイン参加者に限定する

- ステータス: Accepted
- 日付: 2026-07-10
- 関連: #124, ADR-0009, ADR-0010

## コンテキストと課題

イベント参加をキャンセルする API `POST /api/v1/events/{id}/leave`（#124）を追加した。
キャンセルは対象の参加行（`event_members`）を削除し、あわせて参加状態ログ
（`event_participation_logs`）へ `action='leave'` を追記する。

ここで決めなければならないのは、**このエンドポイントが匿名参加者を対象にするか**である。
参加（`POST /events/{id}/join`）側は次の非対称な前提を持つ。

- `join` は**認証任意**（`OptionalAuth`）。ログイン時は `profile_id` を記録し、
  未ログインは匿名参加（`profile_id = NULL`）として `mail_address` のみで識別する。
- 一方 `event_participation_logs.profile_id` は **NOT NULL** 制約のため、
  匿名参加はそもそもログ記録の対象外である（join ログも残らない。#119 で確認済み）。

キャンセルで「どの参加行を消すか」を特定する鍵として何を使うかが分岐点になる。

- **`profile_id`（JWT 由来）で特定** → ログイン参加者しか扱えない。匿名行は消せない。
- **`mail_address`（リクエストボディ由来）で特定** → 匿名参加者も扱えるが、
  誰でも他人のメールアドレスを送れば代理でキャンセルできてしまい、かつ
  `profile_id` が無いため leave ログを残せない（NOT NULL 違反）。

## 決定

**`leave` は認証必須（`RequireAuth`）とし、`(event_id, profile_id)` で参加行を特定して
削除する。匿名参加者（`profile_id = NULL`）はこの API の対象外とする。**

- ルートは `RequireAuth` グループに登録する（`GET /events/{id}/members`、
  `participation-logs` と同じ扱い）。
- 参加者の特定は JWT から得た `profile_id` のみで行い、リクエストボディは受け取らない。
- 削除と leave ログ追記は **同一トランザクション** で原子的に実行する（#119 の join と同方針）。
- 参加していない／イベントが存在しない場合は **404 (`not_found`)** を返す
  （認可チェックを持たない POST 系として ADR-0009 の枠組みに従う）。

匿名参加者のキャンセル手段は本 API では提供しない。必要になった時点で別途設計する
（「検討した代替案」「トレードオフ」を参照）。

## 理由

- **匿名キャンセルには「本人性の担保」と「ログ記録」の両方が原理的に欠ける**。
  匿名行は `mail_address` でしか識別できず、メールを知る第三者による代理キャンセルを
  防げない。さらに `event_participation_logs.profile_id` が NOT NULL のため、
  仮に匿名キャンセルを許しても leave ログを残せず、「leave も記録したい」という
  本 issue の要件（#124）を満たせない。認証必須にすれば本人性とログ記録が同時に満たされる。
- **join のログ方針と一貫する**。#119 で「匿名参加はログ対象外」と決めた以上、
  匿名の leave も本来ログに残せない。ログ記録を伴う状態遷移をログイン参加者に揃えることで、
  `event_participation_logs` は「ログインユーザーの参加状態遷移の履歴」という
  一貫した意味を保てる。
- **識別子を JWT の `profile_id` に一本化でき、実装が単純かつ安全**。ボディで
  `mail_address` を受け取る設計は、なりすまし対策・バリデーション・レート制限など
  攻撃面を増やす。`RequireAuth` 配下で `profile_id` のみを使えば、信頼境界が
  サーバ側 JWT 検証に集約される。
- **ログイン参加者は 1 イベント 1 行**（join の重複チェックが `mail` OR `profile_id` で
  効くため）。`(event_id, profile_id)` で対象を一意に特定でき、削除があいまいにならない。

## 影響（結果）

### 良い影響

- 本人性・ログ記録・実装の単純さを同時に満たす。信頼境界が JWT 検証に集約される。
- `event_participation_logs` が「ログインユーザーの join/leave 履歴」として一貫する。
- handler / repository が join（#119）と同じトランザクション追記パターンに収まり、
  レビューが容易。不存在 404 も ADR-0009 の既定路線で判断コストが無い。

### トレードオフ・制約

- **匿名参加者は API から自己キャンセルできない**。参加は匿名で可能なのに
  キャンセルは不可という非対称が残る。当面の回避策はイベント主催者側での対応
  （主催者向けの参加者削除機能は未実装のため、必要なら別途 issue を起こす）。
- 匿名キャンセルの実需が出た場合は、ログを伴わない削除専用経路（例: ワンタイム
  トークン付きキャンセル URL をメール送付）を別 ADR で設計する必要がある。
  その経路は leave ログを残せない点を明示的に受け入れることになる。

## 検討した代替案

- **`mail_address` をボディで受けて匿名参加者もキャンセル可にする**:
  匿名参加者を救済できるが、(1) メールを知る第三者による代理キャンセルを防げない、
  (2) `profile_id` NOT NULL のため leave ログを残せず #124 の要件を満たせない、
  (3) なりすまし対策・入力検証・レート制限など攻撃面が増える。デメリットが大きく不採用。
- **認証任意（`OptionalAuth`）にして join と対称にする**:
  ログイン時は `profile_id`、匿名時は `mail_address` で分岐する案。分岐が増えるうえ
  匿名側は上記の代理キャンセル・ログ欠落問題を抱えるため、対称性のためだけに
  複雑さと危険を持ち込むことになり不採用。
- **204 No Content を返す**: RESTful だが、追記した leave ログの `createdAt` 等を
  クライアントへ返せない。join が JSON を返すのと対称性を持たせ、
  `LeaveEventResponse`（eventId / profileId / action / createdAt）を 200 で返す方を採用した。
