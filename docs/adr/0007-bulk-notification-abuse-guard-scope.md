# ADR-0007: 一斉送信の濫用対策はレート制限・送信量カウンタで行わない

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: #94, ADR-0001, ADR-0004, ADR-0005

## コンテキストと課題

イベント参加者への一斉送信 API（`POST /api/v1/events/{id}/notifications`）に対する
コードレビューで「認証済み主催者が API を連打でき、参加者への大量送信や Resend の
無料枠（100 通/日）の枯渇を招くため、profileID / eventID 単位のサーバー側レート制限を
入れるべき」という指摘があった。

現状この API にかかっている制御は次の 4 層である。

1. 認証（`RequireAuth`、JWT 必須）
2. 認可（`events.profile_id == profileID`、イベント主催者のみ送信可）
3. 1 リクエストあたりの宛先数上限 `maxNotificationRecipients = 100`（ADR-0005）
4. Resend のレート制限（2 req/秒）に対する指数バックオフ・リトライ（ADR-0005）

決めるべきことは「上記に加えて、profileID / eventID 単位のレート制限や、日次の
累積送信量カウンタを実装するか」である。

### 前提となる構造的事実

宛先上限 `maxNotificationRecipients = 100` は、ADR-0001 で採用した Resend 無料枠の
**100 通/日と同値**である。したがって 100 人規模の正当なイベントで通知を 1 回送ると、
その日の無料枠をちょうど使い切る。これは濫用ではなく本来必要な送信であり、削減できない。

## 決定

一斉送信の濫用・枠枯渇対策として、**profileID / eventID 単位のレート制限も、
日次の累積送信量カウンタも実装しない**。上記 4 層の既存制御で運用する。

無料枠の構造的制約は本 ADR で明文化し、**実運用（参加者への実送信）を始める前に
Resend の有料プランへ移行する**方針とする。プラン移行時に、宛先上限の引き上げや
制御の再検討をあわせて行う。

## 理由

- **日次送信量カウンタは無意味**: 100 人へ通知する正当な必要があるなら 100 通の送信は
  削減不能な本来コストである。カウンタにできるのは送信を安くすることではなく「拒否」だけで、
  「枠が枯れた」を「サーバーが送信を拒否した」に言い換えるにすぎない。利用者から見た結果
  （通知が届かない）は同じで、むしろ正当な送信を自前で止める分だけ悪化する。また無料枠
  100 通/日はイベント 1 回でちょうど使い切るサイズのため、複数用途で枠を融通する余地も無い。
- **真の制約はプラン容量**: 問題はサーバー側ガードの欠如ではなく、無料枠がこの機能に対して
  構造的に小さすぎることにある。正しい解決はコードではなくプラン移行（運用判断）。
- **レート制限（回数/時間）の効果は限定的**: 無料枠 100 通/日は、100 人規模イベントでは
  事実上「1 送信/日」の関所として働く（2 回目の送信は Resend が 429 を返し、ADR-0005 の
  経路で 429 として弾かれる）。profileID / eventID 単位のレート制限が独自に守れるのは
  「小規模イベントでの重複送信」と「ダブルクリック・クライアント再試行暴走による二重送信」
  程度で、費用対効果が見合わない。二重送信の抑止が必要になった場合は、レート制限ではなく
  冪等キーで対処する方が意図に合う。
- **既存の信頼境界で濫用の主経路は塞がれている**: 送信できるのは認証済みかつ当該イベントの
  主催者のみ。他人のイベントの参加者へは送れないため、無差別スパムの経路は存在しない。
  残るリスクは主催者トークンの漏洩だが、これは認証基盤側で扱うべき問題であり、
  レート制限で本質的に解決するものではない。

## 影響（結果）

### 良い影響
- 現段階（宛先最大 100 件・単発操作・無料枠）に対して過剰な機構を持ち込まない。
- 濫用対策コードを入れない根拠が明文として残り、同種のレビュー指摘に一貫して回答できる。
- プラン移行という「本当に効く対策」に判断の焦点が向く。

### トレードオフ・制約
- 小規模イベントでは、主催者が誤って複数回送信すると参加者へ重複メールが届きうる
  （枠が尽きるまでは Resend 側でも弾かれない）。実害が観測されたら冪等キーを検討する。
- 主催者トークンが漏洩した場合の被害範囲は、当該イベントの参加者への送信と 1 日分の
  無料枠に限定される（レート制限が無いため、その範囲内では連打を止められない）。
- **実運用を始める前に Resend 有料プランへ移行することが必須**。無料枠のまま本番投入すると、
  1 イベントの通知で当日の全送信枠を消費し、他用途・他イベントの送信が巻き添えで失敗する。

## 検討した代替案

- **eventID 単位のトークンバケットで連打を抑止**: 既存 `middleware.IPRateLimiter` の構造を
  流用できるが、キーを IP から eventID へ変える改修が要る。守れるのは重複送信程度で、
  無料枠が大規模イベントでは事実上のスロットルになっている以上、費用対効果が見合わず不採用。
  なお `IPRateLimiter` は `c.ClientIP()` をキーにするため「そのまま流用」はできない
  （認証済み濫用対策では profileID / eventID をキーにすべきで、キー関数の作り直しが必要）。
- **日次の累積送信量カウンタで枠を保護**: 上記「理由」の通り、正当な送信を拒否に置き換える
  だけで利用者価値が無く、枠に融通の余地も無いため不採用。正確に作るなら再起動をまたぐ
  永続化（DB）も要り、得られる効果に対してコストが過大。
- **無料枠のまま本番運用する**: 1 イベントの通知で当日枠を使い切り、以降の送信が全滅する。
  実運用に耐えないため、有料プラン移行を前提とする。
