# ADR-0017: イベントキャンセル通知の件名・本文を任意化しサーバー既定文面で補う

- ステータス: Accepted
- 日付: 2026-07-12
- 関連: #141, ADR-0016, ADR-0012
- 変更: ADR-0016 の決定事項 2（cancel API はボディ `{subject, body}` を必須とする）を置き換える（Supersedes, 部分的）

## コンテキストと課題

ADR-0016 でキャンセル通知を Transactional Outbox 化した際、`POST /api/v1/events/{id}/cancel` はリクエストボディ `{subject, body}` を**必須**とした。当時の判断は「通知を送らないより送る方が安全（通知漏れを防ぐ）」であり、必須化はその安全側への倒し方として一貫していた。

一方で運用を検討すると、この必須制約には次の摩擦がある。

1. **参加者 0 人でも文面入力を強制される**: イベント中止の主因が「参加者が集まらなかった」である場合、宛先が 1 件もないにもかかわらず、誰も読まない件名・本文を主催者に書かせることになる。これは頻出ケースであり UX 上の無駄が大きい。
2. **「必須にする」以外の通知保証手段が既にある**: ADR-0016 の Outbox により「キャンセルが成立したら通知は必ず最終的に送られる」ことは構造的に保証済みである。文面が必須かどうかは、この保証とは独立した論点である。

ここで「文面をどう任意化するか」が論点になる。単に任意にすると「通知は予約されるが件名・本文が空」という無内容メールが送られうる。ADR-0016 が守ろうとした「通知漏れを防ぐ」という不変条件を崩さずに、必須制約だけを外す方法を決める必要がある。

## 決定

**cancel API の件名・本文を任意入力に変更し、省略（trim 後に空文字）された場合はサーバーが既定文面を補って必ず outbox に予約する。**

1. `CancelEventRequest` の `subject` / `body` から必須制約を外し、最大長のみ検証する（件名 255 文字以内・本文 10,000 文字以内は据え置き）。空文字は許容する。
2. 補完はサービス層（`EventCommandService.Cancel`）で行う。所有者確認後、trim 後に空であれば既定文面を補う。
   - 既定件名: `【重要】イベントは主催者によりキャンセルされました`（定数 `defaultCancelSubject`）。
   - 既定本文: `対象イベント：` + イベントタイトル。
   - **本文が空のときだけ**イベントタイトルを取得する（件名の既定はタイトル非依存）。取得は所有者確認とは別の軽量クエリ `EventRepository.GetTitle` で行う。
3. Cancel 専用に、空を許容し指定時のみ最大長を検証する `validateOptionalNotificationContent` を新設する。参加者一斉送信（SendBulk）と共用の必須検証 `validateNotificationContent` は変更しない。
4. 補完後は ADR-0016 の経路をそのまま通す。最終的に確定した（常に非空の）subject/body を `CancelWithNotification` に渡し、キャンセル確定と outbox 予約を同一トランザクションで永続化する。Outbox の原子性・非冪等性は維持する。

## 理由

- **「不送信という状態を作らない」を必須制約なしで達成できる**: 文面を任意にしても、省略時は必ず既定文面で予約されるため、通知漏れは起きない。ADR-0016 の安全側の狙いを、UX の摩擦なしに満たせる。
- **フラグ方式より契約が単純**: 「送信要否フラグ」や「参加者数に応じた条件付き必須」も検討したが、前者は「参加者がいても黙ってキャンセルする」経路を許し、後者はフィールドの必須性がサーバー状態（参加者数）に依存して OpenAPI 契約が曖昧になる。任意＋既定補完は、必須性がリクエストボディだけで完結し、参加者数にも送信意図フラグにも依存しない。
- **テンプレートの一元化**: 既定文面をサーバー側に置くことで、将来クライアントが増えても（本 API は OpenAPI から型を共有する前提）文面の品質・一貫性がクライアント実装に散らばらない。
- **層の分離を保てる**: 既定文面の組み立てはサービス層の定数に閉じ、リポジトリの `GetTitle` は純粋なタイトル取得のみとする。業務文言をデータアクセス層に持ち込まない。

## 影響（結果）

### 良い影響

- 参加者 0 人のイベントを、意味のない文面を書かずにキャンセルできる。
- 文面を省略しても通知は必ず既定文面で予約されるため、通知漏れの保証（ADR-0016）は維持される。
- 既定文面がサーバー側に一元化され、クライアント横断で一貫した中止通知になる。

### トレードオフ・制約

- **既定本文は「読んだ時点のタイトル」を使う**: `GetTitle` はキャンセル確定トランザクションの外の別クエリのため、取得からキャンセルまでの間にタイトルが変わると既定本文は旧タイトルになる。タイトルは所有者のみ変更でき、キャンセル操作自体はタイトルを変えないため、実害は無視できる範囲とする（TOCTOU 耐性のための `GetTitle` の `ErrEventNotFound` 分岐も、イベント削除機能が無い現状では実質到達不能な防御的実装）。
- **既定文面は最小限**: 既定本文は `対象イベント：<タイトル>` のみで、日時・場所・主催者からの一言などは含めない。詳細を伝えたい場合は主催者が明示的に文面を入力する運用とする。
- **breaking change ではないが最小ボディが必要**: 全項目を省略する場合でもリクエストボディは最低 `{}` を送る必要がある（完全な空ボディは既存の `bindJSON` が 400 を返す）。ADR-0016 準拠で必須文面を送っていた既存クライアントはそのまま動作する。
- **既定件名の末尾に句点は付けない**（既存のサンプル文言のトーンに合わせた表記判断）。

## 検討した代替案

- **必須のまま維持（ADR-0016 の決定）+ フロントでテンプレートをプレフィル**:
  API 変更ゼロで済むが、テンプレートがクライアント実装に依存し、クライアントが増えると文面がばらつく。参加者 0 人でも文面が必ず送信ペイロードに含まれる。通知保証レベルは本決定と同等だが、一貫性とサーバー責務の観点で不採用。
- **送信要否フラグ（`notify` boolean）を追加し、true のとき subject/body 必須**:
  契約は明快だが、`notify=false` を許すと「参加者がいても黙ってキャンセルする」経路が生まれ、ADR-0016 が守る通知漏れ防止の方針と衝突する。参加者ありのとき false を 400 で拒否する変種は、参加者数カウントとキャンセル確定の間に参加が滑り込む TOCTOU を塞ぐためカウントをトランザクション内に移す必要があり、リポジトリ層が複雑化する。不採用。
- **参加者数に応じた条件付き必須（参加者がいれば必須、いなければ任意）**:
  主催者の「通知漏れを防ぐ」価値観には最も忠実だが、フィールドの必須性がサーバー状態に依存し、静的な OpenAPI スキーマに「必須」と表現できない（`make swag` 生成物と実挙動が乖離する）。フロントもフォーム描画前に参加者数を知る必要がある。不採用。
- **既定文面の補完をリポジトリのトランザクション内で行う**:
  `GetTitle` の別クエリを省け、既定本文が確実にキャンセル時点のタイトルになる利点はあるが、日本語の通知テンプレート文言がデータアクセス層に漏れ、層の分離（handler→service→repository）に反する。既定本文の旧タイトル問題は実害が無視できるため、層の分離を優先して不採用。
