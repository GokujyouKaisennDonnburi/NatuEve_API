# ADR-0008: イベント検索は ILIKE 部分一致＋クエリ時 NFKC 正規化で実装する

- ステータス: Accepted
- 日付: 2026-07-08
- 関連: #104

## コンテキストと課題

イベント検索 API（`GET /api/v1/events?q=...`）を実装するにあたり、検索の照合方式と
その実現手段を決める必要がある。要件は次のとおり。

- 検索対象は 5 フィールド: タイトル(`events.title`)・イベント詳細(`events.description`)・
  主催者名(`profiles.display_name`)・地域名(`events.location`)・持ち物(`event_items.event_item`)。
- キーワードは反復クエリ(`?q=a&q=b`)のリストで受け取り、**キーワード間は AND、
  各キーワードは 5 フィールドを OR で横断**する（AND of ORs）。
- 照合は**部分一致**。大文字小文字を無視し、さらに**半角/全角を同一視**したい
  （全角数字↔半角数字など）。

決めるべきは「この照合を、どのデータ構造・クエリで実現するか」である。現時点の前提として、
`events` の規模は小さく、全文検索基盤（Elasticsearch 等）は導入していない。DB は
PostgreSQL 17（ADR 外の既定構成）。

## 決定

イベント検索は **PostgreSQL の `ILIKE '%キーワード%'` による部分一致**で実装し、
半角/全角の同一視は **クエリ実行時に両辺を NFKC 正規化**して実現する。具体的には:

- WHERE は各キーワードを 1 グループ（5 フィールドの `OR`）とし、グループ間を `AND` で連結する。
  グループ内の各カラムは `normalize(col, NFKC) ILIKE $n` とする。
- キーワードは Go 側で `NFKC 正規化 → LIKE エスケープ(% _ \) → '%'..'%' で囲む` の順に
  組み立て、必ずプレースホルダ経由で渡す（SQL へ直接連結しない）。
- 専用の検索用カラムやインデックス、データ移行は**行わない**（クエリ時正規化のみ）。
- ソート順（`created_at`/`event_date` × `asc`/`desc`）はホワイトリストから固定文字列を選ぶ。

将来データ量が増えて性能が問題になった場合は、本 ADR を起こし直し（`Superseded`）、
生成列＋`pg_trgm` GIN インデックス、または全文検索（`tsvector`）へ移行する。

## 理由

- **要件に対して最小の手段**: 求められているのは「部分一致・大小無視・半角全角同一視」であり、
  `ILIKE` が部分一致と大小無視を、`normalize(_, NFKC)` が半角全角の吸収を、それぞれ標準機能で
  満たす。専用の検索基盤や転置インデックスを持ち込む必要がない。
- **クエリ時正規化はスキーマ・書き込み経路を変えない**: 正規化を保存側に持つ案（検索用カラム＋
  書き込み時正規化＋既存データ移行）に比べ、変更が検索クエリ内に閉じる。座標保護のような
  ドメイン制約（`.claude/rules/geofuzzing.md`）とも無関係で、影響範囲が小さい。
- **正規化の順序で安全性を担保できる**: 全角パーセント `％`(U+FF05) は NFKC で ASCII の `%`
  になり LIKE のワイルドカードになりうる。**NFKC を先・LIKE エスケープを後**に適用することで、
  正規化で生じた記号もリテラルとして無効化される。両辺（カラム・キーワード）を同じ NFKC 形へ
  そろえるため照合基準が一致する。
- **性能劣化は新規に生じたものではない**: `ILIKE '%...%'` は先頭ワイルドカードのため元々
  B-tree インデックスを使えず、対象カラムはシーケンシャルスキャンになる。`normalize(col)` を
  各行へ適用してもオーダーは変わらない（同じ全走査）。現在の規模ではこのコストを許容できる。
- **AND of ORs は素直な SQL で表現できる**: キーワードごとに 5 フィールドの `OR` グループを作り
  `AND` で連結するだけで、意図（全語がどこかに一致）をそのまま SQL に落とせる。持ち物は 1 対多の
  ため `EXISTS` サブクエリで判定し、行の重複を避ける。

## 影響（結果）

### 良い影響
- スキーマ変更・データ移行なしで、検索クエリの変更だけで要件を満たせる。
- 半角/全角の表記ゆれ（全角数字↔半角数字・全角英字↔半角英字・半角カナ↔全角カナ）を吸収し、
  利用者はキーボード入力モードを気にせず検索できる。
- キーワードは常にプレースホルダ経由・ソート順はホワイトリスト固定のため、SQL インジェクション
  の経路を作らない。

### トレードオフ・制約
- **インデックスが効かず全走査**。`events`（および `event_items`）が大きくなると検索レイテンシが
  悪化する。データ量増加時は `pg_trgm` GIN / `tsvector` への移行が必要（その時点で新 ADR）。
- **正規化はクエリ都度**。`normalize(col, NFKC)` を各行・各フィールドに適用するため、全走査に
  関数呼び出しコストが上乗せされる。
- **ひらがな↔カタカナは同一視しない**（NFKC の対象外）。かな統一まで求める場合は別途変換が要る。
- キーワード数には上限（`maxSearchKeywords = 10`）を設ける。AND グループが増えるほどクエリが
  肥大化するため、過剰なグループ生成を防ぐ。

## 検討した代替案

- **検索用カラムに正規化済みテキストを保存し、そこへインデックスを張る**: 書き込み時に NFKC 正規化
  した列を持ち、`pg_trgm` GIN で高速化できる。性能面では優れるが、スキーマ追加・書き込み経路の
  改修・既存データのバックフィル移行が必要で、現在の規模には過剰。将来の移行先として保留し、
  初手では不採用。
- **全文検索（`tsvector` / Elasticsearch 等）**: 大規模・高機能な検索には適するが、部分一致
  （語の途中一致）や記号・数値の扱いが要件とずれやすく、基盤導入コストも大きい。現段階では不採用。
- **アプリ側でのフィルタリング（全件取得して Go で照合）**: DB の負荷は下がらず、ページネーションや
  件数集計と整合させにくい。スケールしないため不採用。
- **正規化をキーワード側だけ／カラム側だけに適用**: 片側だけでは照合基準がそろわず半角全角を
  同一視できない。両辺に NFKC を適用する必要があるため不採用。
