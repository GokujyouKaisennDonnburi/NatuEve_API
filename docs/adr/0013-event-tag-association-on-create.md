# ADR-0013: イベント投稿時のタグ紐づけ設計

- ステータス: Accepted
- 日付: 2026-07-10
- 関連: #120, ADR-0010

## コンテキストと課題

イベント投稿画面では、投稿の前段でタグを作成できる（`POST /api/v1/tags`）。
投稿時にはそれらの作成済みタグを選択してイベントに紐づけたい（#120）。
中間テーブル `event_tags`（`event_id`, `tag_id` の複合 PK、両カラムとも FK・`ON DELETE CASCADE`）は
既に用意されているが、書き込み側の処理が未実装だった。

決めるべき点は以下だった。

1. タグ紐づけの入力をどのエンドポイント・どの形で受け取るか。
2. リクエストに含まれる `tag_id` の検証（形式・重複・存在）をどこで、どこまで行うか。
3. 存在しない `tag_id` が送られてきたときにどう振る舞うか（拒否か無視か）。
4. `event_tags` への書き込みを既存のイベント登録処理とどう整合させるか。

## 決定

**タグ紐づけはイベント投稿 API（`POST /api/v1/events`）のリクエストで `tagIds`（作成済みタグの UUID 配列・任意）として受け取り、既存の Create トランザクション内で `event_tags` へ INSERT する。存在しないタグ ID はエラー（HTTP 400）で拒否する。**

1. `CreateEventRequest` に任意フィールド `tagIds []string`（UUID 配列）を追加する。
   タグ紐づけ専用のエンドポイントは新設せず、イベント投稿の一連の入力に含める。
2. `tag_id` の検証は API サーバー（`service` 層）で行う。
   - 各要素を trim し、空文字は 400 で拒否する。
   - `uuid.Parse` で UUID 形式を検証し、不正な形式は 400 で拒否する（ADR-0010 と整合。ただし本項目は
     複数要素の配列検証のため、パース失敗を検証エラーとして扱う）。
   - trim 後の値で順序を保持したまま重複を除去する。
3. 存在しないタグ ID は **400（`invalid_request` / 「指定されたタグが存在しません」）で拒否する**。
   silent ignore（存在するものだけ紐づけて残りを黙って捨てる）はしない。
4. `event_tags` への INSERT は、既存の `EventRepository.Create` のトランザクション内で
   costs / items / images / pdfs と同じ原子性で行う。
   - INSERT は `ON CONFLICT (event_id, tag_id) DO NOTHING` とし、同一組み合わせの重複を吸収する。
   - FK 違反（PostgreSQL エラーコード `23503`）を検知した場合は、repository 層の
     センチネル `ErrTagNotFound` にラップし、service 層で `ValidationError`（400）へ変換する。

## 理由

- **専用エンドポイントを作らず投稿 API に含める**のは、タグ紐づけがイベント投稿という
  1 つのユーザー操作の一部だから。イベント本体と別トランザクションにすると
  「イベントは作られたがタグ紐づけだけ失敗」という中途半端な状態が生まれ、フロントの
  リトライ設計が複雑になる。
- **検証を service 層に集約する**のは、`CLAUDE.md` の「ドメインロジックの集約／サーバー側が信頼の境界」
  方針に従うため。クライアント側のバリデーションは UX 補助に留める。
- **存在しないタグを 400 で拒否する**のは、silent ignore だとクライアントのバグ（未作成タグの ID や
  他イベントのゴミ ID の送信）に気づけず、意図とずれた紐づけが黙って成立してしまうため。
  早期に失敗させることでデータ整合性を担保し、原因調査を容易にする。
- **FK 違反を存在チェックとして使う**のは、事前に `SELECT` で全 tag_id の存在を確認するより
  クエリ数が少なく、確認と INSERT の間の TOCTOU（確認後に別トランザクションでタグが削除される）も
  避けられるため。DB の制約を信頼の最終境界として使う。
- **`ON CONFLICT DO NOTHING`** は、service 層の重複除去を通り抜けた場合や再送に対する保険。
  複合 PK 違反（`23505`）でトランザクション全体を巻き戻さないようにする。

## 影響（結果）

### 良い影響

- フロントは投稿リクエストに `tagIds` を1つ足すだけでよく、追加の API 呼び出しが不要。
- イベント本体とタグ紐づけが単一トランザクションのため、部分的な失敗状態が発生しない。
- 不正・不存在の tag_id を確実に弾けるため、`event_tags` にゴミデータが入らない。

### トレードオフ・制約

- **エラーメッセージの粒度**: 存在しないタグが複数あっても「指定されたタグが存在しません」と
  まとめて返し、どの ID が不正かは示さない。FK 違反は最初の 1 件で検知するため。
  現状の UX（タグは直前に自分で作成・選択する）では十分と判断した。
- **読み取り側は対象外**: 本 ADR は書き込み（紐づけ）のみを扱う。イベント詳細・一覧レスポンスへの
  タグ表示（`GetByID` 等）は別途対応する。
- **タグ数の上限は設けていない**。極端に多い紐づけを制限する必要が出た場合は
  検証に上限チェックを追加する。

## 検討した代替案

- **タグ紐づけ専用エンドポイント（`POST /api/v1/events/{id}/tags`）を新設する**:
  投稿とは別トランザクションになり「イベントは作成されたがタグ紐づけは失敗」という
  状態が生じる。フロントが 2 ステップを安全に扱う必要が増える。今回は投稿と同時で十分なため不採用。
- **存在しないタグ ID を silent ignore する**: 実装は寛容だが、クライアント側の不整合
  （未作成・ゴミ ID）に気づけず、意図しない紐づけが黙って成立する。#120 の合意により不採用。
- **事前に `SELECT ... WHERE id = ANY($1)` で全 tag_id の存在を確認してから INSERT する**:
  クエリが 1 往復増え、確認と INSERT の間の TOCTOU も残る。FK 違反での検知で十分なため不採用。
- **タグ名（文字列）を受け取り、なければ作成しつつ紐づける（upsert）**:
  タグ作成は投稿前に別 API で完了している前提（#120）であり、投稿 API にタグ作成の責務を
  持たせると座標保護同様に責務が肥大化する。ID を受け取る設計に統一するため不採用。
