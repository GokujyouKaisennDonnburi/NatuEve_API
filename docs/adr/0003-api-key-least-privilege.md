# ADR-0003: 外部サービス API キーは最小権限を原則とする

- ステータス: Accepted
- 日付: 2026-07-07
- 関連: #92, ADR-0001

## コンテキストと課題

外部 SaaS（Resend など）の API キーは、用途を絞らず発行すると、
漏洩時に送信以外の操作（設定変更・ドメイン追加・ログ閲覧など）まで悪用されうる。
本 API は複数の外部サービス鍵を扱っていくため、鍵の権限設計に共通の原則を定めておきたい。
これは特定サービスに閉じない**横断的なセキュリティ方針**であり、
今後追加する他サービスの鍵にも適用する。

## 決定

外部サービスの API キーは**最小権限**で発行する。
Resend の送信キーであれば、**送信専用（Sending access）かつ対象ドメイン限定**とする。

より一般化した原則として、鍵は以下を満たすよう発行する。

- 必要な操作のみに権限を絞る（フルアクセスキーを常用しない）。
- 対象リソース（ドメイン・プロジェクト等）を可能な限り限定する。
- 環境（本番/開発）ごとに別のキーを発行し、使い回さない。

## 理由

- 万一の漏洩時の**被害範囲（blast radius）を最小化**する。
- 送信専用・ドメイン限定であれば、漏洩しても設定改ざんや他ドメインへの悪用ができない。
- 鍵ごとに用途が明確になり、ローテーションや失効の影響範囲を把握しやすい。

## 影響（結果）

### 良い影響
- 漏洩時の被害を送信のみ・特定ドメインのみに封じ込められる。
- 鍵の棚卸し・ローテーションがしやすくなる。

### トレードオフ・制約
- 用途・環境ごとに鍵を分けるため、発行・管理する鍵の本数が増える。
- 権限不足による失敗が起きうるので、必要権限を事前に見極める運用が要る。
- 鍵はコードにハードコーディングせず、`.env`（`.env.example` に変数名のみ記載）で管理する。

## 検討した代替案

- **フルアクセスキーを 1 本で使い回す**: 管理は楽だが、漏洩時の被害が全操作・全ドメインに
  及ぶため不採用。
